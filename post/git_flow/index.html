Git Flow

Git flow is multiple things. On the one side it’s a git extension that you can use to push, pull, merge and checkout. On the other side its a branching model that gives you advice on handling your branching system in git repos.

First I’d like to show the git extensions so that we have that out of the way

Installation of Git Extensions



    MacOS

        Homebrew
            brew install git-flow-avh

        Macports
            port install git-flow-avh

    Linux

        apt-get install git-flow


    Windows

        Cygwin
            wget -q -O - --no-check-certificate https://raw.github.com/petervanderdoes/gitflow-avh/develop/contrib/gitflow-installer.sh install stable | bash

More information on the installation can also be found in the wiki

Usage of the extension

You can start using git-flow by running

git flow init

in the root of your project.

The command will prompt you to answer some questions about the naming of different branches.

Based on multiple sources on GitHub and co, I would advise you to use the default names if your company doesn’t specify different ones.

You can start a new feature with the command

git flow feature start NAME

This command will create a new feature branch that is based on the develop branch. What these branch names mean will be explained in the next part of the entry.

If you are finished with implementing a feature you can run

git flow feature finish NAME

to merge the feature back into develop and remove the branch

Working with others

If you are working with other people you can publish the feature you are working on with

git flow feature publish Name

and others can then pull that feature with

git flow feature pull origin Name

This is just a quick overview of how git-flow handles features. Git flow can also handle releases and hotfixes. But they more or less follow the same principle. Therefore I would urge you to read the wiki on git-flow and maybe try out some of the commands yourself.

Git Flow

Git Flow as a branching model

Branches

When using the git-flow branching model we split the project into the following branches:

  • develop [infinite lifetime]
  • master [infinite lifetime]
  • features [multiple]
  • release branches
  • hotfixes [multiple]

In this system the main branches are:

  • develop
  • master

Where master reflects the current state of release and develop the current state of the finished development. Vincent Driessen also called it the integration branch.

Besides the main branches, we have branches that act in a secondary capacity.

Feature branches

When working on a new functionality/feature we create a new Feature branch for that implementation. These feature branches live amongst the develop branch and merge their implementations into the develop branch when they are finished.

With these multiple features for current releases or future releases can be developed at the same time and merge back into the develop branch when there are finished are needed. This creates a model that enables multiple developers to work on the same project and then merge their features back into the development head.

Please be aware that when merging a feature into the main branch you should use

    git merge --no-ff [branch name]

to create a new commit object. This ensures that no information about the feature is lost and the history is intact.

Release branches

When you and your team have finished implementing a set of features (like the end of a sprint) and tested them (and fixes some bugs). You create a new release

This release then represents the desired point in time you want to separate the current features from the future ones. Like when entering a new sprint. You want to release the implemented features to ensure business value for the client as soon as possible and then work on the new features when this sprints features are released.

The only thing that is now inserted into the release branch are bug fixes.

These are important to ensure that the client gets a working and sound product that can be used in productions.

git checkout -b release-[number] develop

When you have finished fixing bugs in the release branch it is time to push it to production.

So we merge the release branch into the master to reflect the new state of the production product.

We also create a new tag to mark the new release

git tag -a [number]

To keep the bug fixes in the release we have to also merge back the changes into the develop branch.

Hotfix branches

Hotfixes are bug fixes that run between develop and master. They are created when a situation arises that is in need of immediate attention.

These changes, not unlike the release branches, are merged into master and develop. Therefore the creation and merging of branches are the same as with the release branches.
