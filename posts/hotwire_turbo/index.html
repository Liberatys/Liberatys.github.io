Hotwrite / Turbo

Migraton: https://dev.to/coorasse/from-turbolinks-to-turbo-31jl

Drive

A page navigation is seen as a visit which is connected to an â€™actionâ€™. An action can either be [:advance, :replace, :restoration]. If the page in question can be found in cache it will be preloaded to give the appearance of faster loading between frequently visited pages.

If the iOS Adapter or Android adapter are used then the turbo drive will push the new navigiation onto the navigation stack in the native format.

A link can be annotated with a specific action to further customize the change in page load.

<a href="/edit" data-turbo-action="replace">Edit</a>

You can also programmatically trigger a page navigation with a specific action:

Turbo.visit("/edit", { action: "replace" })

Progressbar

Turbo Drive will automatically create a progressbar if a page is loading longer than 500ms. You can change that default by setting

Turbo.setProgressBarDelay(delayInMilliseconds)

Or you can style the bar with scss/css

.turbo-progress-bar {
    height: 5px;
    background-color: green;
}

Asset Tracking

You can reload assets dynamically by giving them the data-turbo-track=â€™reloadâ€™ attribute. The src path for the assets should include some way to identify the version of the asset and make them unique.

Manually trigger a full page reload

You may issue a manual page reload by annotating the page with a meta tag:

<meta name="turbo-visit-control" content="reload">

With this annotation you will skip the cache loading of turbo drive but will restore compatibility to other js libraries that struggle with the way turbo handles navigation.

Redirects after form submission

Turbo Drive expects the server to respond with a 303 HTTP redirect response. An exception are 4xx and 5xx status codes. Trubo drive canâ€™t handle 200 status codes as browsers already have implemented behaviour for the case and turbo canâ€™t hook into the client code from the browser.

Caching

Turbo Drive uses a cache to keep the data for the recently used pages. This is done so that a frequent navigation between two pages is cheaper than having to request the entire application on inital load. It also issues multiple events to the browser upon preparation and loading of the cache

document.addEventListener("turbo:before-cache", function() {
    // ...
})

if (document.documentElement.hasAttribute("data-turbo-preview")) {
    // Turbo Drive is displaying a preview
}

You may also choose to ignore/discard the cache functionality by annotating the page with a html meta tag.

<!-- no-cache disables the caching that is issued by turbo drive -->
<meta name="turbo-cache-control" content="no-cache">

Navigation Events

Like turbolinks, Turbo drive also gives you the option to hook into the load function of turbo. This may be used to start a javascript action after the page has loaded.

document.addEventListener("turbo:load", function() {
    // ...
})

Persisting Elements across page loads

In order to reduce load times and make persisting static information easier, turbo offers a way to persist event listeners and state for an element. Thus drive will restoe the state and all the attached javascript listeners of the object and ensures that your element is ready to go.

<div id="cart-counter" data-turbo-permanent>1 item</div>

Frames

Turbo Frames are created by wrapping a section of a page with a trubo-frame element. This allows turbo to register a section of the page and manage the content if the server sends information to the specific channel-id.

<turbo-frame id="message_1">
  <h1>My message title</h1>
  <p>My message content</p>
  <a href="/messages/1/edit">Edit this message</a>
</turbo-frame>

You may have multiple turbo-frames on the same page, but they should have different ids.

You can replace the content of a turbo-frame by sending new content from the server to the client. Thus allowing you to dynamically replace / set content for a section of the page. This works perfectly for forms and other submission elements akin to forms.

The message can then be broadcasted from the server side:

Turbo::StreamsChannel.broadcast_action_to(
  @stream_id,
  action: @action,
  target: @target,
  content: @content
)

Lazy Loading

Frames can loaded lazyily by providing a src where the content may be found.

<turbo-frame id="set_aside_tray" src="/emails/set_aside">

</turbo-frame>

You can also provide an inital content for lazy loaded elements. An example would be a spinner or some other progress display.

<turbo-frame id="set_aside_tray" src="/emails/set_aside">
  <img src="/icons/spinner.gif">
</turbo-frame>

In / Out of frame

Navigation in a frame will by default only effect the section/frame which the link or navigation element is included in. If you want to handle navigation which also affects change outside of the frame you can annotate the element with:

target="_top"

The opposite is also possible. What if you want to only change a frame with a click on a link outside of the frame? Then you can use the data-turbo-frame attribute which will only effect the frame specified by the given.

<form action="/messages/1/delete" data-turbo-frame="message_1">
  <input type="submit" value="Delete this message">
  (with a confirmation shown in a specific frame)
</form>
