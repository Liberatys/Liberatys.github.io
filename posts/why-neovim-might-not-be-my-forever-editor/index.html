Why [Neovim|Vim] might not be my forever editor

07.03.2021
Coding , Tools

A relationship with an expiration date?

I’ve been using Neovim(and Vim before that) for about three years now. A teacher of mine told me to play the vim adventure game so that I had something to do in class. It was not love at first sight.

And for many reasons:

  • There were no fancy images / emojis
  • No refactoring tools
  • It was running on the terminal
  • Strange keybindings
  • Modes? Why can’t I just insert characters in normal mode?
  • Bad Java integration(which was the main language used in school)
  • No way to know how to exit on the first try!

and the list should go on.

Using Neovim

After some time I came back to vim and started to tweak it into a semi-IDE. I had spent hours searching for extensions on GitHub, reading through Reddit and StackOverflow threads and watching hours of youtube videos of people writing text in a terminal? (Upon reflection that seems somewhat strange :D).

After hours of configuration I, finally, had a tool that allowed me to quickly jump into any file and edit a small portion, jump to another file and remove some code… It was great! The great parts were not writing code.. but editing code. To this day I believe that both vim and neovim are the best tools for editing code(don’t shoot the messenger). The moment you understand how vim commands are structured and you have some muscle memory you unlock a very powerful code editor.

And after moving to Neovim, I even had async completion of code, more plugins(now written in both VimL and Lua). And as the power of my editor grew, so did my configuration files.

I spent many hours rewriting old configurations and removing/adding plugins just so that I could keep up the momentum I had gained. It was a time of great happiness.

Vim as my daily driver at work

After three years of school, I started my internship as a software developer. Development is mainly done with Ruby on Rails with some Javascript on the side. So naturally, the company was using RubyMine as the main editor for development. A tool, developed by Jetbrains, which gives you pretty much everything to develop any kind of ruby application.

During my first week, I had the pleasure to have a pair programming session with a co-worker and was puzzled by his editing speed! Code seemed to appear in the editor out of nowhere!

After that experience, I struggled with Neovim as my daily driver as I had witnessed the power of RubyMine. So for one week after that, I switched to RubyMine and tried to adjust to the change. I had moved from a modal editor to an IDE. A change that I struggled with! No more mode switching, no more visual mode to quickly move around code or run some operations on a selected area. No grammar associated with editing code.

After that week of RubyMine, I switched back to Neovim and didn’t look back for a long time. I added some plugins and focused on optimizing my workflow and got faster at editing code very fast! It’s always easier to work towards a goal than just to work for the void!

What made me rethink my choice

Now, 8 months into my internship, I more and more realise that software development is a lot more than just writing some code.

A big portion of development is:

  • Testing
  • Refactoring
  • Code quality / Linting

RubyMine and editors like VSCode have very easy integration and setup for these use cases. Click on a button and run the tests for the current method. Click and refactor the current block into a method. Click and drag and easily edit multiple lines at once. Search and replace for the entire project? Yeah sure why not!

Most of these functionalities can be configured and installed into neovim by loading some plugin or writing some small script to do the same thing. But then again, I have to spend hours just catch up with the stuff that RubyMine comes preinstalled with. And because Ruby is a very dynamic language, I was never really able to configure Neovim to match the autocompletion and documentation lookup that RubyMine offers.

And yes, I also have to configure VSCode, but from what I’ve seen, tried, and read I have to conclude that VSCode has a much easier configuration with some very sane defaults and an easy extension system.

Conclusion

Whilst writing this small entry I became aware of how much time I had spent configuring vim and neovim. I’ve been an active advocate of neovim(which can be attested to by my co-workers). But I now somewhat struggle with the everlasting tweaking I do on my editor. I would rather focus on writing, testing and refactoring code than on having to configure my editor just to match the features provided by any modern IDE.

Strangely enough, writing this article has made me very sad :/. I’ve considered neovim to be my best coding buddy. And writing down my struggles with it somewhat betrays the many hours we have spent debugging and tweaking code together.

For now, Neovim will stay my editor of choice. But I plan on finding a suitable replacement soon. I think either Emacs or RubyMine(or IntelliJ) are strong contenders. I categorically rule out VSCode as I’m not a big fan of Electron as a framework or javascript/typescript as a language.

Emacs by the way I also an editor I would have to heavily configure in order to unlock the features I need. But from what I read it seems like Emacs could be much more than just an editor. It could become the one-trick pony for all my operations and development.

In the end, I need an editor that has both good support for Rust and Ruby. As I mainly use Ruby at work and Rust for everything outside of work.

I’d be happy to hear about your editor of choice and maybe advice on how to change my development environment.

Cheers! Have a great day!
